# Docusaurus Setup

This project publishes repo docs through Docusaurus.

You do not need to scaffold a new Docusaurus site for this repo. The site already exists in `website/` and is wired to render the markdown in `Docs/`.

## Source Of Truth

- Source markdown: `Docs/`
- Docusaurus app: `website/`

Do not author long-form project docs in `website/docs`. That scaffold is intentionally unused here.

## One-Time Local Setup

1. Make sure Node.js 20+ and npm are installed.
2. Open PowerShell in the repo root.
3. Run:

```powershell
Set-Location website
npm install
```

That installs the Docusaurus app dependencies only. Your actual docs still live in `../Docs`.

## Daily Preview Workflow

Run:

```powershell
docs-tools start
```

Then open:

```text
http://localhost:3000/docs/
```

What happens next:

- Docusaurus starts a local dev server attached to the current terminal.
- Editing files in `Docs/` updates the site preview.
- You see live stdout/stderr directly in that terminal.

When you are running in the attached mode, stop it with normal terminal interruption such as `Ctrl+C`.

If you want the old tracked detached mode instead, run:

```powershell
docs-tools start --background
```

Stop the tracked background dev server with:

```powershell
docs-tools stop
```

`docs-tools start` is the repo-local wrapper around `website/`'s `npm start`. By default it stays attached so you can watch the server logs. Use `--background` when you want a detached tracked server instead.

Use `docs-tools status` to see whether the tracked background server is still running and where its logs are. Use `docs-tools doctor` if the local docs workflow feels broken and you want a quick prerequisite check.

## What To Edit

- Edit page content in `Docs/`.
- Create ordinary pages and sections with `docs-tools new-page` and `docs-tools new-section`.
- Control sidebar order with `sidebar_position` on docs and `_category_.json` metadata on folders.
- Treat `_category_.json` as the section marker; a section README is optional.
- Reorder existing pages and sections with `docs-tools reorder <TargetPath> <Position>`.
- If you omit `-Position`, `docs-tools` assigns the next available position automatically.
- Only touch `website/sidebars.ts` when the autogenerated docs shell itself needs to change.
- Change site-wide branding, footer, navbar, or deploy metadata in `website/docusaurus.config.ts`.

## Production Build Check

Use this before merging docs-navigation changes or site-config changes:

```powershell
docs-tools check
```

`docs-tools check` validates docs metadata, catches common docs-site mistakes, and then runs the Docusaurus production build. If you specifically want the raw website commands, `docs-tools build` wraps `npm run build`.

## Other Docusaurus Commands

`docs-tools` forwards the `website/package.json` scripts directly, so these all work from the repo root:

```powershell
docs-tools build
docs-tools clear
docs-tools deploy
docs-tools serve
docs-tools swizzle
docs-tools write-translations
docs-tools write-heading-ids
docs-tools typecheck
docs-tools docusaurus <args...>
```

## Current Wiring

- The Docusaurus docs plugin reads from `../Docs`.
- Blog output is disabled.
- Coding-standard `Current/` and `Templates/` are excluded from the site build.
- `website/sidebars.ts` delegates to autogenerated navigation rooted at `Docs/`.
- Section ordering comes from `_category_.json` and page ordering comes from doc front matter.

## Optional VS Code TOC Bridge

If you want new pages and sections to start with a generated table of contents:

1. Install the VS Code extension `Markdown All in One`.
2. Run:

```powershell
docs-tools install-bridge
```

After that, `docs-tools new-page` and `docs-tools new-section` will queue Markdown All in One's `Create Table of Contents` command through the local bridge extension. If either dependency is missing, `docs-tools` falls back to plain scaffolding and skips TOC generation.

## Deployment Notes

- Update `website/docusaurus.config.ts` `url` and repository metadata before a real deployment target is chosen.
- Keep docs deployment separate from gameplay feature branches when possible.
- Validate `npm run build` before merging navigation changes.

## When To Update The Site App

- Changing global navigation
- Adjusting theme, branding, or footer content
- Changing deploy metadata or base URL
- Changing how the autogenerated docs shell works

## Common Mistakes

- Do not write the real project docs under `website/docs`.
- Do not edit `website/.docusaurus/`, `website/build/`, or `website/node_modules/`.
- If a new doc page exists but does not show in navigation, check `_category_.json`, `sidebar_position`, and the folder location under `Docs/`.
- If the build fails on links, fix the markdown links in `Docs/` rather than weakening the Docusaurus checks.
- If `docs-tools check` reports an unprocessed TOC marker, either install the optional bridge workflow or remove the marker and author the TOC manually.
