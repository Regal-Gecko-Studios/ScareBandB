# Authoring Docs

Write docs in `Docs/`. Keep the site shell in `website/` thin.

## Authoring Rules

- Prefer concrete examples tied to this repo.
- Put commands in fenced code blocks.
- Use repository-relative paths.
- Add front matter when a page needs a stable slug or custom title.
- Use `docs-tools new-section` and `docs-tools new-page` for routine scaffolding instead of hand-creating the same front matter repeatedly.
- Keep normal docs navigation data in the `Docs/` tree with `_category_.json` and `sidebar_position`; `website/sidebars.ts` is only the thin autogenerated shell now.

## Content Rules

- Replace stale process docs instead of layering contradictory notes on top.
- Keep Confluence references clearly marked as legacy context only.
- Avoid TODO-only pages. Either write the policy or do not create the page yet.
- Keep operational runbooks close to the scripts they describe.

## Linking Guidance

- Link sibling docs with relative markdown links inside `Docs/`.
- Link repo files with explicit repo-relative paths when giving commands or examples.
- Prefer stable section headings over loose “see above” references.

## New Section And Page Workflow

Create a section:

```powershell
docs-tools new-section GameDesign -Title "Game Design" -Position 9
```

Create a page inside that section:

```powershell
docs-tools new-page GameDesign Fear-Loop -Title "Fear Loop" -Position 2
```

Create a top-level docs page without a section:

```powershell
docs-tools new-page Setup -Title "Setup"
```

What those commands handle for you:

- create the markdown file
- generate front matter and canonical slugs
- write `_category_.json` for new sections
- treat `_category_.json` as the section marker; `README.md` is only the optional landing doc
- keep the new content compatible with autogenerated Docusaurus sidebars
- support both top-level `Docs/<Page>.md` pages and `Docs/<Section>/<Page>.md` pages

If you omit `-Position`, `docs-tools` now assigns the next available sidebar position automatically instead of reusing the same default for every new page or section.

## Reordering Existing Docs

Use `docs-tools reorder` when you want to move an existing page or section and shift its siblings automatically.

Examples:

```powershell
docs-tools reorder Art-Source 4
docs-tools reorder GameDesign 3
docs-tools reorder GameDesign/Fear-Loop 2
```

What it updates:

- page `sidebar_position` values
- section `_category_.json` `position` values
- sibling positions in the same parent container so you do not have to hand-edit every item below the move target

## Optional TOC Automation

`docs-tools` can optionally ask VS Code to run Markdown All in One's table-of-contents command after scaffolding a new page or section.

Setup:

1. Install the VS Code extension `Markdown All in One`.
2. Run:

```powershell
docs-tools install-bridge
```

Notes:

- If both the bridge and Markdown All in One are available, new pages and sections include a temporary TOC marker and queue a request for VS Code to run `Create Table of Contents`.
- If either dependency is missing, `docs-tools` skips TOC generation automatically.
- Use `-NoToc` when you want to suppress TOC generation even when the optional bridge is available.

## Local Site Commands

Use the repo-local wrappers instead of dropping into `website/` for normal work:

```powershell
docs-tools start
docs-tools stop
docs-tools status
docs-tools check
docs-tools doctor
docs-tools serve
docs-tools write-heading-ids
docs-tools docusaurus <args...>
```

`start` now runs the Docusaurus dev server attached to the current terminal so you can see live stdout/stderr. Use `docs-tools start --background` when you want the old detached tracked-server behavior. `stop` and `status` only apply to that tracked background mode. `doctor` checks common local prerequisites such as `node`, `npm`, `website/node_modules`, and the optional VS Code TOC bridge.

Use `docs-tools help <command>` for the detailed option list on any command:

```powershell
docs-tools help new-page
docs-tools help new-section
```

For advanced Docusaurus metadata, the scaffold commands support both typed flags and generic passthrough fields:

- `new-page`: use `-Field key=value` and `-FieldJson key=json` for any extra front matter.
- `new-section`: use `-DocField` / `-DocFieldJson` for the section `README.md`, and `-CategoryField` / `-CategoryJson` for `_category_.json`.
- Use `-LinkType generated-index` for autogenerated landing pages, or `-CategoryJson 'link={...}'` when you need full manual control of the category link object.
